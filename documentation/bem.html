BEM methodology

While we were not using this methodology on our past projects Sitecore is excellent place to start using it and getting use to it and all new sitecore projects from now on should practice it. Drupal 8 already implemented SMACSS which is one of those popular methodologies about which i will not get into details but you are free and encouraged to google them up.


So what exactly is BEM. To prevent any confusion we will just paste the definition from their official site. BEM (Block, Element, Modifier) is a component-based approach to web development. The idea behind it is to divide the user interface into independent blocks. This makes interface development easy and fast even with a complex UI, and it allows reuse of existing code without copying and pasting.


This is the official BEM documentation


Long story short main purpose of BEM is to create semantical and clean code that can be reused. General idea which we were have been following on the Sitecore is to consider Sitecore a big Lego building castle which contains a lot of those little lego blocks. We can build it anyway we like and we can put them anywhere we want and all the blocks can be different sizes and there can be unlimited amount of them.

Considering we are building a "lego castle" we need a dynamic way to build Blocks, elements and modifiers (which BEM is a shortcut for) so we can reuse them anywhere we want.


Organism or Block

  • Block in BEM is a functionally independent page component that can be reused. In HTML, blocks are represented by the class attribute. lego-block. In our folder structure that would be 03_organism folder. In there we would put general styles for the blocks like paddings, spacings, backgrounds, specific element stylings and modifiers for that block.


  • Element is a composite part of a block that can't be used separately (i completely disagree with this one and i will explain further down the line) from it as we are all aware each organism is built out of molecules and by following that philosophy as we have might guessed it each block will be built out of elements. In our folder structure the element related styles would go in 02_molecule if it is a standalone element or in 03_organism folder if elements are composite parts of its parent. Elements are separated from Blocks or Organism's by double underscore __


    Best example here would be to ask yourself does my block have a specific title, button of any other element for that matter inside of it. If that is the case the second question to ask ourselves is: are those elements the same throughout the site or are they block specific? If they are block specific then you have a composite element and you can go inside the 03_organism/and make a scss file named _o-block-lego.scss..


    Molecule or Element

    Molecule or Element can be used seaprately or as a part of organism please review 03_organism folder and the code in it, but it does not have to as we can see from 02_molecule folder

    It is compeltely up to the developers as We have already noted to decide which parts of the build can stand as standalone entities or composite entities.

    Everything is neatly organised and we at every time know what is where. Sitecore tends to get really big and elements tend to repeat themselves so it is really important to before starting to work on a project to decide which elements are reusable and decide which styles are reusable. and structure your project accordingly

Atom or modifier

Atom or Modifier is an entity that defines the appearance, state, or behavior of a block or element

In atoms we should put general appearances, hover or other states so we can reuse them. This can be reviewed in both 02_molecule and 03_organism folders

Important!

There is a certain prerequisite of using this kit when we download it or clone it via github into the sitecore project and that is we need to run a command which will install all our dependencies for us and set everything up for us. This needs to be run inside wherever the package.json is located. Package json looks like this . If we inspect the screenshot we will see that we have Dev dependecies. These are actually all the packages used currently. From minification to compilation to gulp itself. To install everything with one liner is to just run npm install where that package.json is located and you are ready to work.

If for any reason you get the error that you are missing bindings which usually happens because your environment has changed since running `npm install`. Run `npm rebuild node-sass` to build the binding for your current environment.
